# NeuroDrop — Полное техническое задание

> Сервис продажи доступа к подпискам зарубежных цифровых сервисов для аудитории РФ.
> Этот документ — единый источник правды для разработки. Если что-то противоречит другим источникам — приоритет у этого файла.

---

## 0. TL;DR для разработчика (Claude / Cursor)

**Что строим:** премиальный B2C интернет-магазин цифровых подписок (ChatGPT Plus/Pro, Cursor Pro, Claude Pro, Gemini Advanced, Perplexity Pro, YouTube Premium, Spotify, Grok, Canva Pro, Replit Core, TryHackMe и др.) для пользователей из РФ.

**Стек:** Next.js 15 (App Router) + TypeScript + Tailwind v4 + shadcn/ui + Drizzle ORM + PostgreSQL + Better-Auth + Lava.top + Resend + Socket.IO + grammy (Telegram) + Selectel.

**Ключевая логика бизнеса:** у каждого товара несколько вариантов покупки (продление существующего аккаунта / готовый аккаунт с подпиской / другие). Для "продления" покупатель оставляет логин/пароль через зашифрованную форму. Админ обрабатывает заказ вручную в админке, общается с клиентом через встроенный чат + Telegram.

**Перед началом кода:** прочитай этот документ целиком, задай уточняющие вопросы по архитектуре и спорным местам, нарисуй план первой итерации (MVP), дождись одобрения, и только потом пиши код.

## 1. Контекст и бизнес-цель

### 1.1. Проблема

Российские карты не принимаются большинством зарубежных цифровых сервисов. Пользователи вынуждены искать обходные пути: иностранные карты, посредников в Telegram, серые сервисы без гарантий.

### 1.2. Решение

NeuroDrop — белый, профессиональный сервис-посредник:

1. Покупатель выбирает сервис (например, Cursor Pro) и вариант (продление своего аккаунта / готовый аккаунт).
2. Оплачивает в рублях через российские платёжные методы (карта/СБП/крипта).
3. Получает доступ к подписке: либо его аккаунт продлевается, либо ему выдаются данные нового аккаунта.
4. Может в любой момент связаться с поддержкой через личный кабинет или Telegram.

### 1.3. Ценности бренда

- **Доверие** — UX как у банковского приложения, прозрачность, отзывы, гарантии.
- **Скорость** — выдача за минуты после оплаты для готовых аккаунтов.
- **Поддержка** — быстрая, человеческая, через привычный канал (Telegram + сайт).

### 1.4. Целевая аудитория

- Возраст 18–45 лет.
- IT-специалисты, фрилансеры, креативщики, студенты, дизайнеры, маркетологи.
- Уровень технической грамотности — выше среднего, но интерфейс должен оставаться простым.

### 1.5. Название и тональность

- **Название:** NeuroDrop (рус. НейроДроп). Ассоциация: быстрая выдача нейросетевых сервисов.
- **Тон:** уверенный, технологичный, без панибратства. Не "хайпово-молодёжно", а "профессионально-современно".

## 2. Юридический и платёжный контекст

### 2.1. Статус заказчика

- **Самозанятый** на старте.
- Лимит **2.4 млн ₽/год** — нужно отслеживать в админке.
- Онлайн-касса по 54-ФЗ **не нужна**, чеки автоматически через "Мой налог".
- Принимаем оплаты **только от физлиц**.
- Архитектурная цель: подготовить миграцию на ИП/УСН без переписывания кода (через 3–6 месяцев работы).

### 2.2. Платёжные шлюзы

| Приоритет | Шлюз                     | Роль                                                       |
| --------- | ------------------------ | ---------------------------------------------------------- |
| #1        | **Lava.top**             | Основной — карты, СБП, поддержка самозанятых, авточеки ФНС |
| #2        | **Robokassa**            | Резерв — на случай проблем с Lava.top                      |
| #3 (опц.) | **CryptoCloud**          | USDT TRC20 / TON для тех, у кого нет карт                  |
| Будущее   | **ЮKassa** / **Т-Касса** | Подключим после открытия ИП                                |

**Архитектурное требование:** все платёжные провайдеры должны реализовывать единый интерфейс `PaymentProvider` (методы: `createPayment`, `getStatus`, `refund`, `verifyWebhook`). Подключение нового шлюза = одна реализация интерфейса, без изменений в бизнес-логике.

### 2.3. Юридические страницы (обязательны)

- Публичная оферта.
- Политика конфиденциальности (152-ФЗ).
- Согласие на обработку персональных данных.
- Реквизиты в футере: ИНН самозанятого, контакты.
- Cookie-баннер (можно простой, не GDPR-уровня).

## 3. Инфраструктура и хостинг

### 3.1. Решение

- **Хостинг:** Selectel (Россия, доступен из РФ без VPN).
- **VPS:** Ubuntu 22.04 LTS, минимум 2 vCPU / 4 GB RAM / 40 GB SSD на старт.
- **БД:** Selectel Managed PostgreSQL 16 с автобэкапами.
- **Файловое хранилище:** Selectel Object Storage (S3-совместимое) — логотипы, аватары, файлы из чата.
- **Reverse proxy:** Caddy (автоматический Let's Encrypt) — проще nginx, всё из коробки.
- **Деплой:** Docker Compose. CI/CD — GitHub Actions, билд образа → push в registry → SSH-деплой на VPS.

### 3.2. Домен

- Покупаем на reg.ru. Рекомендуемые варианты: `neurodrop.ru`, `neurodrop.online`, `neurodrop.shop`.
- Резервный домен (на случай блокировок) — короткий .online или .shop.

### 3.3. Email

- **Resend** — основной провайдер транзакционных писем.
- Резерв на будущее: Unisender / SendPulse (российские).
- Все шаблоны писем — настраиваются в админке (текст, заголовок).

### 3.4. Мониторинг и аналитика

- **Yandex.Metrica** с включённым Вебвизором и настроенными целями.
- **Sentry** для ошибок (бесплатный тариф достаточен).
- Логи приложения — JSON-лог в stdout, ротация средствами Docker.
- Uptime-мониторинг — UptimeRobot (бесплатно).

### 3.5. Безопасность

- HTTPS обязательно.
- Cloudflare Turnstile или Yandex SmartCaptcha на формах.
- Rate limiting на login/register/checkout (через middleware).
- Шифрование чувствительных данных в БД (AES-256-GCM, ключ из ENV).
- Логирование любого доступа к расшифрованным данным аккаунтов в `audit_log`.
- Регулярные бэкапы БД (daily на Object Storage).

## 7. Функциональные требования

### 7.1. Публичная часть

#### 7.1.1. Главная `/`

- **Hero:** заголовок-оффер, подзаголовок, поиск по сервисам (autocomplete), CTA "Перейти в каталог".
- **Сетка популярных сервисов** — 8–12 карточек с логотипами и ценой "от".
- **"Как это работает"** — 3–4 шага с иконками и описанием.
- **Преимущества** — 4 блока (скорость, гарантия, поддержка, оплата в ₽).
- **Социальные доказательства** — счётчики (заказов, отзывов, рейтинг), карусель отзывов.
- **FAQ** — 6–10 вопросов (раскрывающиеся).
- **CTA-блок** перед футером.

#### 7.1.2. Каталог `/services`

- Сетка карточек по категориям (AI, Видео/Музыка, Дизайн, Разработка, Образование, Прочее).
- Фильтры: категория, цена (range), тип (продление / готовый аккаунт), наличие.
- Поиск с подсветкой совпадений.
- Сортировка: популярные / дешёвые / дорогие / новинки.
- Бесконечная прокрутка или пагинация.

#### 7.1.3. Страница услуги `/services/[slug]`

- **SEO-оптимизирована:** H1 "Купить [Service] [Plan] в России", уникальный description, schema.org `Product` + `Offer` + `AggregateRating` + `Review`.
- **Хлебные крошки.**
- **Галерея** (логотип крупно + скриншоты сервиса опционально).
- **Tab/Select-выбор тарифа** — длительность (1/3/6/12 мес) и вариант (продление / готовый аккаунт). Цена пересчитывается мгновенно.
- **Описание услуги** — что включено, какие функции открывает подписка.
- **Гарантии** — иконки (мгновенная выдача, гарантия замены, поддержка 24/7).
- **FAQ** — частые вопросы по конкретному сервису.
- **Отзывы** — только от подтверждённых покупателей, с фильтром по рейтингу.
- **Похожие услуги** — карусель.
- **Sticky CTA** на мобилке "Купить за X ₽".

#### 7.1.4. Корзина и Checkout

- Корзина опциональна на старте (можно сразу с страницы товара в checkout с одной позицией). На фазу 2 — полноценная корзина.
- **Checkout:**
  - Email + телефон (если не залогинен — создаётся аккаунт автоматически с magic link).
  - **Динамическая форма** под выбранный вариант (см. 7.4).
  - Поле промокода.
  - Согласие с офертой (чекбокс, обязательный).
  - Кнопки оплаты: "Картой/СБП через Lava", "Криптой" (если активно).
  - При нажатии — создание заказа со статусом `pending_payment`, редирект в шлюз.
  - На возврате `/orders/[id]/success` — показ статуса с polling/SSE до подтверждения вебхуком.

#### 7.1.5. Личный кабинет `/account`

- **Мои заказы** — список со статусами, возможность открыть детали.
- **Детали заказа** — что куплено, оплата, выданные данные (если есть), чат с поддержкой.
- **Чат с поддержкой** — встроенный realtime-виджет на странице заказа.
- **Профиль** — редактирование имени, телефона, привязка Telegram (по deeplink в бот).
- **Отзывы** — оставить отзыв на завершённый заказ.
- **История платежей** — скачивание чеков (PDF из ФНС).

#### 7.1.6. Чат с поддержкой

- **Realtime через Socket.IO.**
- **Один тред на каждый заказ** + отдельный общий тред "до покупки" для предпродажных вопросов.
- **Уведомления админу в Telegram** с кнопкой "Открыть в админке" + inline-возможностью ответить прямо из бота.
- **Email-нотификация** покупателю при ответе админа (если нет активного коннекта).
- **Поддержка вложений** — изображения и PDF до 10 МБ, хранятся в Object Storage.
- **Индикатор "печатает".**
- **История за всё время.**

#### 7.1.7. Контентные страницы

- **Блог `/blog`** — карточки постов, пагинация, категории.
- **Пост `/blog/[slug]`** — markdown с подсветкой кода, автор, дата, schema.org `Article`, поделиться, related posts.
- **О нас** `/about`.
- **Контакты** `/contacts` — Telegram, email, форма обратной связи.
- **Юридические:** `/legal/offer`, `/legal/privacy`, `/legal/terms`.

### 7.2. Админ-панель `/admin`

Доступ — только для `role: 'admin' | 'manager'`. Для дизайна — отдельный layout (sidebar + topbar), не похожий на публичную часть, но в той же стилистике.

#### 7.2.1. Дашборд

- **KPI-карточки:** выручка (день/неделя/месяц), кол-во заказов, средний чек, конверсия checkout, активные чаты, остаток лимита самозанятого.

### 7.3. Telegram-интеграция

#### 7.3.1. Бот для админа (обязательно)

- **События для уведомлений:** новый заказ, успешная оплата, новое сообщение в чате, новый отзыв, ошибка платежа.
- **Формат:** информативное сообщение + inline-кнопки ("Открыть в админке", "Ответить").
- **Двусторонний чат:** ответ на сообщение покупателя из Telegram (через reply на сообщение бота) → публикуется в чат сайта от имени поддержки. Источник сообщения помечается как `telegram` в БД.
- **Команды:** `/stats` — быстрые KPI за день, `/orders` — последние заказы.
- Бот настраивается через ENV: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_ADMIN_CHAT_ID` (можно несколько через запятую для команды менеджеров).

#### 7.3.2. Бот для покупателя (фаза 2, опционально)

- Покупатель может привязать свой аккаунт сайта к Telegram (через deeplink с одноразовым кодом).
- Получение уведомлений о статусе заказа в Telegram.
- Возможность писать в поддержку через бот — сообщение попадает в общий тред.

### 7.4. Динамические формы покупателя

Это ключевая особенность. Каждый вариант товара (`product_variants.form_schema`) имеет свой набор полей, которые покупатель заполняет на checkout.

**Формат `form_schema` (JSON):**

```json
{
  "fields": [
    {
      "key": "login",
      "label": "Логин или email от аккаунта",
      "type": "text",
      "placeholder": "example@gmail.com",
      "required": true,
      "encrypted": true
    },
    {
      "key": "password",
      "label": "Пароль",
      "type": "password",
      "required": true,
      "encrypted": true
    },
    {
      "key": "2fa_backup",
      "label": "Резервные коды 2FA (если включена)",
      "type": "textarea",
      "required": false,
      "encrypted": true
    },
    {
      "key": "notes",
      "label": "Дополнительные комментарии",
      "type": "textarea",
      "required": false,
      "encrypted": false
    }
  ]
}
```

**Поддерживаемые типы:** `text`, `password`, `textarea`, `email`, `select` (с options), `checkbox`.

**Поля с `encrypted: true`** — шифруются клиентским и серверным AES-GCM перед сохранением в БД, хранятся в `order_items.customer_input_encrypted`. Расшифровка только в админке, с записью в `audit_log`.

**Конструктор формы в админке** — drag-and-drop UI для редактирования `form_schema` без JSON.

### 7.5. SEO-требования (приоритет — Яндекс)

#### 7.5.1. Технические

- 100% SSR на публичных страницах.
- Динамический `generateMetadata` для каждой страницы товара/категории/поста.
- **Schema.org разметка** через JSON-LD:
  - `Organization` на всех страницах.
  - `BreadcrumbList` на вложенных страницах.
  - `Product` + `Offer` + `AggregateRating` + `Review` на страницах товаров.
  - `FAQPage` на страницах с FAQ.
  - `Article` на постах блога.
- **`/sitemap.xml`** — автогенерация (товары + категории + блог + статичные).
- **`/robots.txt`** — разрешить всё кроме `/admin`, `/account`, `/api`.
- **IndexNow API** — пинг Яндекса при изменении товара/поста.
- **Open Graph** + Twitter Cards.
- **Канонические URL.**
- **Hreflang** не нужен (только ru).

#### 7.5.2. Производительность

- LCP < 2.5s, CLS < 0.1, INP < 200ms на мобиле (4G).
- Картинки в `next/image` с AVIF/WebP fallback.
- Lazy loading для всего ниже первого экрана.
- Шрифты через `next/font` с display swap.
- Preload критичных ресурсов.
- Service Worker не нужен на старте.

#### 7.5.3. Контент

- **URL-структура:** `/services/[slug]` — короткие, человекочитаемые слаги (`cursor-pro`, `chatgpt-plus`, а не `cursor-pro-1-month-renew`).
- Под разные тарифы товара — `?plan=12m` query, не отдельные URL (чтобы не размывать вес).
- **Title-шаблон:** `Купить {Product} {Plan} в России — оплата картой РФ | NeuroDrop`.
- **H1 на страницах товара:** содержит ключевик "Купить".
- **Уникальные description** для каждого товара.
- Внутренняя перелинковка через "Похожие услуги" и упоминания в блоге.

#### 7.5.4. Аналитика

- **Yandex.Metrica** с настроенными целями:
  - `view_product` — просмотр товара
  - `add_to_checkout` — переход в checkout
  - `payment_started` — нажатие "Оплатить"
  - `payment_success` — оплачено
  - `review_submitted` — отзыв оставлен
- Вебвизор включён (но без записи checkout-полей с чувствительными данными — настроить маски).
- E-commerce события (товар, корзина, покупка).

### 7.6. Email-уведомления (через Resend + React Email)

Шаблоны (все настраиваемые в админке):

1. **Подтверждение регистрации** (magic link / verify email).
2. **Сброс пароля.**
3. **Создание заказа** (с реквизитами оплаты).
4. **Подтверждение оплаты** + чек ФНС.
5. **Заказ в обработке** (для продлений — после получения данных).
6. **Заказ выполнен** (с выданными данными для готовых аккаунтов).
7. **Новое сообщение в чате** (если пользователь оффлайн).
8. **Запрос отзыва** (через 3 дня после выдачи).
9. **Возврат оформлен.**

Все письма — в едином стиле бренда, мобильно-адаптивные.

## 13. Правила для Claude (как работать в этом проекте)

### 13.1. Подход

- **Думай прежде чем писать.** Перед началом каждого крупного куска — короткий план в чат.
- **Спрашивай о реальных трейдоффах.** Не спрашивай очевидное.
- **Делай красиво.** Это лицо бизнеса.
- **Каждая итерация — рабочий артефакт.** После окончания этапа должно быть что показать в браузере.

### 13.2. Кодстайл

- TypeScript strict, никаких `any`.
- Zod-валидация на границах (API, формы, ENV).
- Серверная логика — в `src/server/services/*`, не в route handlers напрямую.
- Компоненты — функциональные, hooks-only.
- `'use client'` — только там где реально нужно (state, effects, browser API).
- Импорты через aliases (`@/components`, `@/lib`).

### 13.3. Комментарии в коде

- **Не** комментировать очевидное (`// импорт модуля`, `// рендер компонента`).
- **Комментировать** только бизнес-логику, неочевидные ограничения, ссылки на документацию шлюзов.
- JSDoc для публичных функций библиотек (`src/lib/`).

### 13.4. Git

- Conventional Commits (`feat:`, `fix:`, `refactor:`, `chore:`, `docs:`).
- Атомарные коммиты, понятные сообщения.
- Pull Request на каждую крупную фичу (даже если разработка соло).

### 13.5. Безопасность

- **Никогда** не логируй пароли, токены, расшифрованные данные клиентов.
- **Всегда** валидируй входные данные через Zod.
- **Всегда** проверяй авторизацию и роль перед чувствительными операциями.
- **Никогда** не клади секреты в репозиторий — только через ENV.

### 13.6. Тестирование

- Не пиши тесты до явной просьбы — но критичные функции (шифрование, расчёт суммы заказа, проверка вебхуков) покрой Vitest.
- E2E (Playwright) — только основной flow покупки, перед запуском.
